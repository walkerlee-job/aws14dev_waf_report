## 5. 異常事件分析

### 事件 A：管理後台存取嘗試（16:00）

- **規則**：`block-admin-from-external-ip`（Priority 11）
- **觸發條件**：請求 URI 符合 `/wp-admin`、`/wp-login.php` 或 `/wp-json/wp/v2/users`，且來源 IP 不在內部 IP 清單中
- **次數**：1 次（16:00 台北時間）
- **評估**：單次嘗試，可能為一般性 WordPress 探測掃描。雖屬正常 WAF 攔截範圍，但值得記錄。如持續增加需提升警覺。

### 事件 B：IP 信譽封鎖（17:00、23:00）

- **規則**：`AWSManagedIPReputationList`（managed-reputation-ip，Priority 2）
- **次數**：4 次（17:00 ×2、23:00 ×2）
- **評估**：來源 IP 被 AWS 全球情報庫列入黑名單。17:00 為尖峰流量時段，同時出現信譽封鎖較值得關注；23:00 的深夜封鎖可能為境外自動掃描。

### 事件 C：Bot 來源資料中心信號（16:00）

- **規則**：`SignalKnownBotDataCenter`（BotControl）
- **次數**：1 次（16:00 台北時間）
- **評估**：來自已知 Bot 資料中心，與 AdminBlock 同時發生（16:00），可能為同一來源的探測行為。

### 事件 D：大規模 Bot 流量（持續全天）

- **規則**：`SignalNonBrowserUserAgent`（45 次）
- **分布**：幾乎 24 小時均有分布，14:00 與 17:00 最多
- **國家**：主要來自日本（JP 43/45，約 96%）
- **評估**：該流量模式與自動化爬蟲或程式性 HTTP 客戶端（如 Go HTTP library、curl 等）一致。Label 維度中可見 `curl`、`go_http`、`http_library`、`okhttp` 等 Bot 識別標籤，確認為非瀏覽器自動化工具。現行 `OverrideAction: Count` 配置讓 Bot 偵測僅計數，最終因無 Allow 規則而遭 Default Action 封鎖，但此流量並未被明確的 Bot 封鎖規則攔截。

## 6. 共通性分析

### 6.1 攻擊類型分析

**主要威脅：非瀏覽器自動化存取（80.4%）**

- 跨 CloudWatch 維度交叉驗證：`LabelName=non_browser_user_agent`、`LabelName=http_library`、`LabelName=go_http`、`LabelName=curl`、`LabelName=okhttp` 均有出現在 BlockedRequests 指標，確認流量來自程式化 HTTP 客戶端
- 流量分布均勻（每小時 1～4 次），非集中突波，符合長期持續掃描行為
- 封鎖來源以日本 IP 為主，但 Bot 本身性質（程式化客戶端）不代表攻擊意圖，可能為 SEO 爬蟲、監控工具或研究掃描

**次要威脅：IP 信譽 / 匿名化基礎設施（14.2%）**

- AWSManagedIPReputationList（4 次）+ HostingProviderIPList（4 次）共 8 次
- 匿名代理或雲端主機出口 IP，可能為跳板行為
- VulnerabilityCategory 維度交叉確認：`AWSManagedIPReputationList` 顯示 4 次、`HostingProviderIPList` 顯示 3 次，與 ManagedRuleGroupRule 維度接近一致

**特定攻擊類型（0 次）**

- Attack=KnownBadInputs：0（未偵測 Log4Shell、SSRF 等已知惡意輸入）
- Attack=XSS：0（未偵測跨站腳本攻擊）
- CrossSiteScripting_QUERYARGUMENTS：0
- ReactJSRCE_BODY：0
- ExploitablePaths_URIPATH：0
- WordPressExploitablePaths_URIPATH：0

> **結論**：目前沒有明確的 Injection、RCE 或 XSS 攻擊跡象，封鎖以 Bot 流量管理為主。

### 6.2 時間模式

- 封鎖事件隨流量增減呈正相關：14:00 最高流量（447 允許），封鎖也最多（9 次）
- SignalNonBrowserUserAgent 全天均勻分布，表示為持續性 Bot 掃描，非人為手動操作
- IP 信譽封鎖集中在下午至深夜（17:00、23:00），可能與境外惡意 IP 活躍時段相關

### 6.3 日本 IP 異常高比例

- 43/56 封鎖來自日本 IP（76.8%），而 SignalNonBrowserUserAgent 的封鎖分布與總封鎖幾乎完全吻合
- 可能原因：
  1. 日本 CDN/雲端業者出口 IP 被 Bot 使用（假冒日本 IP）
  2. 真實日本 Bot 程式定期掃描
  3. 正常日本服務的 API 整合客戶端（非攻擊，但不符合 WAF 瀏覽器 UA 規範）
- **建議**：調閱 WAF Sampled Requests 確認實際 User-Agent 字串，判斷是否為已知爬蟲

## 7. 建議事項

### 短期建議（立即～2 週）

1. **調閱 WAF Sampled Requests**  
   進入 AWS Console → WAF → go-acl-waf → Sample requests，查看 SignalNonBrowserUserAgent 的實際 User-Agent 字串，確認是否為已知合法爬蟲（如 Googlebot、Bing 等）或惡意工具。

2. **確認 managed-bot-control 的適當 Override 設定**  
   目前 `managed-bot-control` 設為 `OverrideAction: Count`，使 Bot 流量僅被標記而非直接封鎖，最終靠 Default Action（Block）處理。若確認非瀏覽器 UA 全數應封鎖，可考慮改為 `OverrideAction: None`（使用 BotControl 本身的 Block 行為），以便獲得更精確的規則層級封鎖指標。

3. **觀察 16:00 管理後台存取事件**  
   查看 Sampled Requests 確認來源 IP 是否為已知測試環境或外部掃描，決定是否需要加入 IP 黑名單或強化告警。

### 中期建議（2 週～2 個月）

4. **設定 CloudWatch Alarm**  
   針對 BlockedRequests（Rule=ALL）設定每小時閾值告警（建議閾值：>20 次/小時），確保異常封鎖突波能即時通知。

5. **評估 BotControl 管理策略**  
   目前 Bot 流量以 Default Action 封鎖，缺乏明確的 Bot 規則封鎖記錄。建議：
   - 使用 `BotControl Targeted` 模式（需額外費用）獲得更精細的 Bot 行為分析
   - 或在 BotControl 後增加明確的 Block 規則，針對 `awswaf:managed:aws:bot-control:signal:non_browser_user_agent` 標籤封鎖

6. **日本 IP 深度分析**  
   使用 CloudFront 存取日誌（若已啟用）分析日本 IP 的實際請求路徑，區分：
   - 正常業務 API 客戶端（需加入白名單或修正 UA 設定）
   - 惡意掃描（需加入 IPSet 黑名單）

7. **檢視匿名 IP 政策**  
   `managed-anonymous-ip` 設為 Count，`anonymous-ip-captcha` 對 `AnonymousIPList` 設為 Captcha，但 `HostingProviderIPList` 子標籤無特定規則。確認 VPN/雲端主機出口 IP 的適當處理方式。

### 長期建議（2 個月以上）

8. **建立 WAF 日報機制**  
   自動化本報告流程（使用 CloudWatch Logs Insights 或 Athena 查詢 WAF Logs），每日產生 Markdown 報告存入 S3，結合 Lambda + EventBridge 定時觸發。

9. **啟用 WAF 完整日誌**  
   若尚未啟用，設定 WAF Logging 至 Kinesis Data Firehose → S3，以便進行 IP 層級、URI 層級的精細分析（目前 CloudWatch Metrics 僅提供彙總統計，無法查看個別請求詳情）。

10. **速率限制規則調優**  
    目前 `request-rate-breaker`（300 req/5min）與 `request-rate-captcha`（700 req/5min）均為 Count 或 Captcha 模式，在 56 次封鎖中無速率封鎖觸發記錄。確認速率閾值是否符合實際業務需求，必要時降低閾值以強化 DDoS 防護。

| `WebACL=go-acl-waf, VulnerabilityCategory={Cat}` | 查詢漏洞類別交叉驗證 |

### 資料限制

- CloudWatch Metrics 為彙總統計，無法查看個別請求 IP / URI / User-Agent
- 建議啟用 WAF Logging（Kinesis Firehose → S3）以取得請求層級詳細資料
- `ManagedRuleGroup` 維度（無 ManagedRuleGroupRule）若 OverrideAction 為 Count，BlockedRequests 可能為 0
- `ManagedRuleGroupRule` 維度的 BlockedRequests 表示「該子規則匹配且請求最終遭封鎖」，封鎖動作可能來自後續規則或 Default Action
- 國家維度出現於 `list-metrics` 但查詢 BlockedRequests 為 0 的國家（RU、SG、TW 等），可能有允許流量通過但無封鎖事件

### 查詢步驟

1. `aws wafv2 list-web-acls --scope CLOUDFRONT --region us-east-1` 確認 ACL 存在
2. `aws cloudwatch list-metrics --namespace AWS/WAFV2 --metric-name BlockedRequests` 確認實際 Dimension 組合
3. 批量查詢所有 Rule、ManagedRuleGroupRule、Country、Attack、VulnerabilityCategory 維度
4. `aws wafv2 get-web-acl` 取得完整 ACL 規則設定
5. `aws cloudwatch get-metric-statistics --metric-name AllowedRequests` 取得允許請求量作為基準
